# Feature 002: Config and Policy Model

## Summary
Define repository-local behavior in `forge.toml`, covering hook enablement, tool execution settings, and commit message policy.

## Motivation
A static binary is not enough; teams need a declarative way to express local policy without editing source code.

## Data Model
- Root: `[hooks]`
- Hook: `[hooks.<name>]`
  - `enabled = true|false`
  - `tools` map for executable tools
  - optional `policy` section for hook-specific behavior
- Commit message policy:
  - `conventional_commits`
  - `append_ticket_footer`
  - `require_ticket`
  - optional `branch_pattern`

## Functional Requirements
1. Loader must resolve config in this order:
   1. `FORGE_CONFIG` (absolute or repo-relative)
   2. `forge.toml` in repository root
2. Missing config should produce actionable output (`forge init`).
3. Tool settings must support extension filtering, include/exclude patterns, file passing, per-file execution, restage behavior, and failure strategy.

## Validation Requirements
- Fail fast on malformed TOML.
- Empty hooks map should not crash.

## Compatibility
- Starter config generated by `forge init` should be immediately runnable for JS/TS repos.

## Out of Scope
- JSON/YAML config parsing in v1.
- Remote config includes.
